# 来店・売上予測AI（小売・サービス業 ポイントシステム）

小売・サービス業の **ポイントシステム** の購買情報を中心に、天候などを組み合わせて
**来店予測（主）→ 売上予測（従）** を行う分析AIプロジェクトです。

実データが揃う前に、**擬似データ** で「良い精度の分析ができるか」を検証し、
分析の型（プロンプト／分析パイプライン）を確立することを当面の MVP とします。

詳細な仕様・論点は [`要件定義.md`](./要件定義.md) を参照してください。

### 実データソース（提供テーブル定義より）

| 区分 | テーブル | 役割 |
|---|---|---|
| 購買（ポイントシステム） | `CSPM_BUYING_HISTORY` / `_DETAIL` / `CSPM_PRODUCT` / `CSPM_SHOP` | 説明変数（客単価・カテゴリ・会員等） |
| 来店（通行人カウントAI） | `FLOW_COUNTER_EVENT_DATA`（入退室イベント） | **目的変数**（来店客数） |
| 天候 | `FLOW_COUNTER_DAILY_WEATHER` / `_WEATHER_LOC`（OpenWeatherMap One Call v3） | 説明変数 |

> ※ 当初「CPSS」と表記していたが正しくは **CSPM**。
> **SNS** と **外部の人流予測データ** は提供テーブルに含まれないため、実データ版では当面対象外（要判断）。
> 天候は Daily Aggregation（実績寄り）のため、翌日予測には別途 **予報値** の取得が必要。詳細は要件定義書を参照。

### フェーズ6 成果物（実データ）
- `src/real_data_etl.py`：`test_data/` のTSVを読み込み、購買を **店舗×日**（購買客数・純売上・客単価・会員数・返品）、通行人を **端末×日**（入店数）に集計。実行は `python src/real_data_etl.py`。
- `phase6_real_eda.ipynb`：カバレッジ確認・時系列・曜日/祝日効果・来店↔売上相関の可視化。
- 主な確認結果：連続日次60日以上の店舗 **86/187**、来店(receipts)↔純売上の相関 **0.91**、会員数↔来店 **0.995**、曜日・祝日効果あり（祝日は平日比 約0.80）。返品(RETN)は購入額が負値のため純売上計算で調整済み。

### フェーズ6.5 成果物（実データ予測・プール方式）
- `src/real_visit_forecast.py`：86店プール（2024Q4・91日）で購買来店数・純売上を予測。`test_end` で検証期間を指定可能。
- `phase6_5_real_forecast.ipynb`：検証期間比較・学習・可視化・特徴量重要度。

**検証期間別の来店MAPE（14日間・参考値）**

| 検証期間 | 来店MAPE | 検証後半MAPE | 備考 |
|---|---|---|---|
| 10月後半（10/18〜31） | 15% | 9% | 通常期 |
| 11月後半（11/17〜30） | **8%** | 2% | 通常期・最も良好 |
| 12月前半（12/2〜15） | **8%** | 1% | デフォルト検証期間 |
| 12月後半（12/17〜30） | 29% | **47%** | 年末急落で悪化 |

> 以前の「後半ほど外れる」現象は **12/28〜30 の年末特殊期** が主因。通常期では擬似データに近い精度。

**疎な101店舗（連続日次でない店舗）の活用**

「連続日次60日以上」という制約を外し、全187店舗を学習に使う2方式を追加し、同じ検証期間（12月前半）で比較。

| 方式 | 対象店舗数 | 学習行数 | 来店MAPE |
|---|---|---|---|
| 従来：密店舗のみ（min_days=60） | 86 | 7,147 | 7.8% |
| B案：カレンダーのみモデル（ラグなし） | 187 | 9,944 | 8.7% |
| A案：直近実績ラグ＋空白日数（gap_days） | 187 | 9,757（**+37%**） | 9.8% |

- 精度低下はごくわずかで、店舗網羅性を大きく広げられる。特徴量重要度は `shop_id_code`・`dow` が支配的で `gap_days` 自体の寄与は小さい（店舗×曜日の傾向で大半を説明できてしまうため）。
- 精度重視なら**B案**、疎店舗のラグ情報も活かしたいなら**A案**が候補。関数：`train_calendar_only_model` / `train_asof_model` / `compare_coverage_strategies`。

## フェーズ2・3の結果（参考値）

検証期間（直近90日）の精度（擬似データ・seed=42）：

| 予測 | 手法 | MAPE |
|---|---|---|
| 来店客数 | 勾配ブースティング | 約9% |
| 来店客数 | ベースライン（曜日平均／前年同日） | 約21〜29% |
| 売上 | 来店予測 × 客単価予測 | 約10% |
| 売上 | ベースライン（曜日平均） | 約23% |

来店予測がベースラインを大きく上回り、「来店×客単価」の分解方式が売上の直接予測よりも良い精度を確認。

### フェーズ7（任意）：継続学習・施策最適化の方針

定期実行の中で「使うほど賢くなる」仕組み。用途で適した技術が分かれる。

| 用途 | 適した技術 | 補足 |
|---|---|---|
| 予測精度の向上 | 定期再学習＋ドリフト検知（＝強化学習ではない） | 予測は教師あり学習のため |
| 施策提案の改善 | 強化学習／**文脈付きバンディット** | クーポン最適化などに適合 |

- 前提：**施策の実行結果（報酬）を記録する効果測定ログ**が必要（フェーズ6と連携）
- 推奨ステップ：効果ログ基盤 → 再学習ループ → A/B・効果測定 → 文脈付きバンディット → 本格RL
- 詳細は [`要件定義.md`](./要件定義.md) の「9.5 フェーズ7」を参照
