# 来店・売上予測AI 要件定義書

## 1. 背景・目的

小売・サービス業の **ポイントシステム** に蓄積される購買情報を中心に、天候・人流・SNS などの外部情報を組み合わせて、
**来店予測（主）→ 売上予測（従）** を行う分析AIを構築する。

本ドキュメントは、その第一歩となる **MVP（最小限の実用プロトタイプ）** の要件を定義する。

### ゴール

- 実データが揃う前に、**擬似データ** を使って「良い精度で来店・売上予測ができるか」を検証する
- そのための **分析の型（プロンプト／分析パイプライン）** を確立し、後で実データに差し替えられる状態にする
- データベース連携・自動化は後続フェーズとし、まずは **分析の中身** を固める

### 既存の技術検証（PoC）の位置づけ

- 前段として、イベント・観光分野を題材にした **翌日売上予測の技術検証（`demo.ipynb`）** を実施済み。
- 合成データに埋め込んだ傾向を機械学習が学習し、単純ベースライン比で MAPE 約10%（ベースライン約31〜33%）を達成。
「擬似データ → 特徴量 → 学習 → 予測 → 精度評価」の一連が機能することを確認済み。
- 本要件では、この検証済みの型を **小売・サービス業ポイントシステム／来店予測** に作り替える。

## 2. 対象ドメインと予測対象


| 項目      | 内容                     |
| ------- | ---------------------- |
| 対象ドメイン  | 小売・サービス業（ポイントシステム運用企業） |
| 主たる目的変数 | **来店予測**               |
| 従たる目的変数 | **売上予測**（来店予測から接続）     |


### 予測の粒度（決定：両方／段階的に実施）

来店予測には大きく2つの捉え方があり、**両方を採用**する。設計が異なるため段階的に進める。


| 区分                 | 内容                          | 主な用途         | 相性の良い説明変数       | 実施      |
| ------------------ | --------------------------- | ------------ | --------------- | ------- |
| **A. 店舗単位・日次（主軸）** | 各店舗の1日の来店客数を予測 → 客単価を介して売上へ | 人員配置・発注・販促計画 | 天候・人流・SNS・カレンダー | フェーズ2・3 |
| **B. 会員個人単位**      | 会員ごとの次回来店・離反（churn）を予測      | クーポン配信・CRM施策 | CSPM購買履歴・RFM    | フェーズ4   |


> まず **A（店舗単位・日次来店客数 → 売上）** を主軸に構築し、型が固まった後に **B（会員個人）** へ展開する。

### 売上予測への接続方法（暫定）

- 売上 ＝ 来店客数 × 客単価 と分解し、来店予測に客単価モデル（曜日・天候・会員構成で変動）を掛けて売上を推定する方式を採用予定。
- 直接「売上金額」を予測する方式とも比較し、精度の良い方を採る。

## 3. MVP の定義（決定）

「まずしっかり分析できるプロンプト等の作成をMVPとする」という方針を、
**「分析パイプライン（ノートブック）＋ LLMによる解釈・施策提案プロンプト」** までを MVP として具体化する。

- **MVPに含む**
  - 擬似データの生成（購買・来店・会員・天候・人流・SNS）
  - 来店予測モデル（機械学習）＋ 精度検証
  - 来店 → 売上の接続
  - 分析・予測結果を **解釈し施策提案するLLMプロンプト**（自然言語での説明・打ち手提示）
- **MVPに含まない（後続）**
  - データベース連携・ETL
  - 定期自動実行・ダッシュボード
  - 実データ接続

> ※ 補足：精度の中核は機械学習が担い、LLMプロンプトは「結果の解釈・説明・施策提案」を担う役割分担とする。
> 「プロンプトのみで予測精度を出す」構成ではない点を明記（プロンプトは分析パイプラインを駆動・解釈する位置づけ）。

## 4. スコープ

### 含めるもの

- 現実的なパターンを埋め込んだ擬似データ生成
- 特徴量エンジニアリング（データリーク対策込み）
- 来店予測モデルの学習・予測・精度評価（ベースライン比較）
- 来店 → 売上予測への接続
- 予測結果を解釈・説明し、施策を提案するLLMプロンプトの設計

### 含めないもの（後続フェーズ）

- 外部API・DBからの実データ取得・連携
- 定期自動実行・運用基盤・ダッシュボード化
- 会員個人単位の離反予測（オプション、第4フェーズ）

## 5. 説明変数（データソース）設計

予測時点で入手可能な情報のみを使用する（**データリーク対策**）。

### 5.0 実データ確認結果（テストデータ実物のEDA）★重要

`test_data/` に提供された実データ（TSV／タブ区切り）を確認した結果、当初の想定と異なる重要点が判明した。

#### 提供物の実態

| データ | ファイル | 期間 | 規模 | 主な単位 |
|---|---|---|---|---|
| 購買履歴 | `buying_data/buying_history.csv` | 2024-10-01 〜 2026-06-15 | 2,439,380 伝票 / **187 店舗** / 約 40.7 万会員 | 伝票（RCPT 241.3万・RETN 2.6万） |
| 購買明細 | `buying_data/buying_history_detail.csv` | 同上 | 約 745.6 万明細 | 商品明細 |
| 商品マスタ | `buying_data/product.csv` | - | 約 18.6 万商品 | 商品（大/中/小分類あり） |
| 通行人カウント | `flow_counter_data/flow_counter_event_data.csv` | 2025-11-19 〜 2026-06-12 | 53,381 イベント / **端末は実質1台**（`1000000`）+ `UNKNOWN` 6,005件 | 入退室イベント（IN/OUT） |
| 天候 | （実物データ未提供・**定義のみ**） | - | - | - |

- **店舗×日のユニーク数（購買）＝ 9,946**。日次×店舗パネルとして扱える。
- **通行人カウント（端末1台のIN）＝ 140日分・IN計25,176・平均約180人/日**。単店の来店実測として扱える規模。
- 購買の `D_TRANSACTION_DT` は時刻が `00:00:00` 固定＝**日付粒度のみ**（POSが時刻を渡していない）。時間帯分析は購買側では不可。
- 通行人カウントの `D_EVENT_DT` は `YYYYMMDDHHMMSS`＝秒単位。単店の**時間帯分析は可能**。

#### ★最重要の論点：来店（通行人）データと購買データが「店舗単位でリンクできない」

- 購買は **187 店舗**、通行人カウントは **端末1台のみ**で、端末→店舗の対応表（`FLOW_COUNTER_DEVICE` / 店舗マッピング）が提供されていない。
- そのため「ある店舗の通行人来店数」と「同じ店舗の購買」を突き合わせる（＝来店→購買のコンバージョン分析）ことが、現データでは **できない**。
- 期間も、購買 2024-10〜、通行人 2025-11〜 と**開始がずれる**（重複は 2025-11〜2026-06 の約7か月）。

> これは以前ご相談いただいた「購買と来店がリンクした本番データがない」懸念が、実データで顕在化したもの。
> **ロジック（パイプライン）は正しく作れば、後で正しく紐づいたデータを入れて活用できる**が、現データでは両者を直結した精度検証はできない。

#### そこで、実データフェーズの目的変数を次のように整理する（方針転換）

| 系統 | 目的変数 | データ | 位置づけ |
|---|---|---|---|
| **A. 購買来店・売上（主軸／リンク良好）** | 店舗×日の **購買客数（伝票数/会員数）** と **純売上** | CSPM 購買（187店・良質にリンク） | 実データで精度検証する本命。天候（要実物）・カレンダー・ラグで予測 |
| **B. 通行人来店（単店・独立デモ）** | 端末1台の **日次入店数（IN集計）** | FLOW_COUNTER（単店） | 「通行人→来店」パイプラインを独立に実証。店舗マッピングが来たら A と接続 |

> ※「純粋な通行人来店（店前通過含む全来店）」は B でのみ観測でき、「購買に至った来店」は A で観測できる。
> 両者を結ぶ**コンバージョン**は、端末→店舗マッピング入手後（フェーズ6.5）に扱う。

### 5.1 提供データソース（実テーブル定義に基づく）

共有されたテーブル定義により、実データのソースが判明した。
当初「CPSS」と表記していたが、正しくは **CSPM**（※会員ID自体は `CPSS...:USR` 形式）。また「人流予測」「SNS」の想定は実データと差異があるため下記に修正する。

#### ① CSPM 購買情報（ポイントシステム / POS連動）

| テーブル | 内容 | 連携 |
|---|---|---|
| `CSPM_BUYING_HISTORY` | 標準の購買履歴。**伝票単位**（POS連動）。売上(RCPT)・返品(RETN)を含み、取消されていないもののみ送られる | 差分（直近5日） |
| `CSPM_BUYING_HISTORY_DETAIL` | 購買履歴の明細（購入商品ごと） | 差分（直近5日） |
| `CSPM_PRODUCT` | 標準の商品マスタ（オプション扱いのため全レコードがあるとは限らない） | 全件 |
| `CSPM_SHOP` | 標準の店舗マスタ（同上） | 全件 |

- 日次に集計して説明変数化：客単価・買上点数・カテゴリ別売上構成・会員来店数/比率・RFM 等。
- **注意①**：返品(RETN)を差し引いて純売上・純客数を算出する。
- **注意②**：差分連携（直近5日）のため、運用は増分更新を前提に設計する。
- **注意③**：会員個人粒度（フェーズ4）は伝票・明細の会員IDで CSPM 内で完結。

#### ② 来店データ（通行人カウントAI / FLOW_COUNTER）★目的変数の出所

| テーブル | 内容 | 連携 |
|---|---|---|
| `FLOW_COUNTER_EVENT_DATA` | 通行人カウントAIが検知した**入退室イベントの生データ** | 差分（直近5日） |

- これを日次集計したものが **来店客数（目的変数B）** になる。実測カウントのため代理指標は不要。
- **実データの実態**：端末は実質1台（`1000000`）。IN 28,100 / OUT 25,281 と数が一致せず、`UNKNOWN` 端末が 6,005 件ある。日次はIN集計で約180人/日、データのある日数は140日（期間207日中）で**欠損日がある**。
- **集計ルール（要確定）**：来店数は原則 `C_EVENT_TYPE = IN` の日次カウントを採用。`UNKNOWN` 端末の扱い（除外/合算）、欠損日補完、営業時間の扱いを決める必要がある。
- **要確認**：この端末がどの店舗（CSPMの187店のどれか）に対応するか＝**端末→店舗マッピング**（現状不明）。

#### ③ 天候データ（FLOW_COUNTER_DAILY_WEATHER）

| テーブル | 内容 | 連携 |
|---|---|---|
| `FLOW_COUNTER_DAILY_WEATHER` | 気象データ。**OpenWeatherMap One Call API v3 の Daily Aggregation** のレスポンスに基づく | 差分（直近5日） |
| `FLOW_COUNTER_WEATHER_LOC` | 上記の集計対象地点 | 全件 |

- 気温・降水などを来店予測の説明変数に利用。列は雲量・湿度・降水量・気圧・最低/最高/時間帯別気温・風速/風向・生JSONペイロード。
- **実データの実態**：今回の提供物には**天候の実データCSVは含まれず、テーブル定義のみ**。実データ検証には別途エクスポートが必要。
- **注意（リーク）**：Daily Aggregation は基本「実績（その日の集計）」。翌日予測に使うには **予報値** が必要。
  同 API の forecast を別途取得する設計にする（実績天候を当日特徴に使うと本番でリーク）。

#### ④ 人流（外部の独立した「人流予測」データは提供テーブルに無い）

- 当初想定した「店舗周辺の人出予測（Agoop 等の外部データ）」は提供テーブルに含まれない。
- 来店予測の文脈では、FLOW_COUNTER は**来店そのものの実測**であり、当日値を特徴量に使うのは循環/リーク。
  → **過去の来店ラグ・移動平均**として利用する（独立した先行指標ではない）。

#### ⑤ SNS情報（今回の提供テーブルには無し）★要確認・要判断

- 共有されたテーブルに SNS のソースは含まれていない。
- 現状の擬似データは SNS を「翌日来店の先行指標」として使っているが、**実データでは未提供**。
- 選択肢：(a) 別途 SNS 取得手段を用意する／(b) 当面 SNS なしで構築する／(c) 将来追加する。

### 5.2 追加提案データ（あると精度が上がる）

小売・サービス業／ポイントシステムで効きやすい変数を提案する。採否は相談のうえ決定。

- **カレンダー系**：曜日、祝日、連休、月初/月末、給料日・年金支給日、学校休暇
- **販促系**：チラシ配布日、特売・セール、クーポン配信/利用、キャンペーン、ポイント◯倍デー
- **価格系**：主要カテゴリの価格・値引き率
- **会員系**：会員数推移、会員ランク構成、入会経過月、来店間隔の分布
- **競合・地域系**：近隣競合のセール、地域イベント・祭り、スポーツ開催
- **マクロ系**：季節性、長期トレンド、物価・消費マインド

> 特に **ポイント◯倍デー・クーポン配信・特売・給料日** は来店に直結しやすく、優先的に検討したい。

## 6. 擬似データ仕様


| 項目     | 内容                                            |
| ------ | --------------------------------------------- |
| 期間     | 2年分（約730日）                                    |
| 粒度     | 日次 × 店舗（**MVPはまず1店舗**）                        |
| 主テーブル  | 来店客数・売上・客単価 / 天候 / 人流 / SNS / カレンダー・販促 / 会員集計 |
| 埋め込む傾向 | 曜日周期・季節性・祝日/給料日効果・天候影響・人流連動・SNS先行効果・販促効果・ノイズ  |

> ※ 現行の擬似データは上記の想定スキーマ。実データ（CSPM / FLOW_COUNTER）接続時（フェーズ6）に、
> 提供テーブルの列・粒度へ合わせて読み込み・日次集計処理を実装する。SNS・外部人流は提供データに無いため、
> 実データ版では当面これらの特徴量を外す前提（採否は「確認事項」参照）。


> 擬似データは「現実に起こりうる関係性」を意図的に埋め込み、モデルがそれを学習・再現できるかで妥当性を検証する。

## 7. モデル・評価設計


| 項目       | 内容                                                   |
| -------- | ---------------------------------------------------- |
| メインモデル   | HistGradientBoostingRegressor（勾配ブースティング系。多特徴量・欠損に強い） |
| 比較ベースライン | 前年同日 / 直近移動平均 / 曜日平均                                 |
| 将来検討     | LightGBM、Prophet（季節性）、来店×客単価の階層モデル                   |


### 評価設計

- **時系列分割**（過去で学習 → 未来で検証。ランダム分割は禁止）
- 指標：MAE / RMSE / MAPE ＋ **繁忙日（来店上位）に絞った誤差**
- ベースライン比で機械学習が有意に良いことを確認
- 特徴量重要度で「何が来店を動かすか」を解釈

## 8. 処理フロー

```
1. 擬似データ生成（購買/来店・天候・人流・SNS・販促・会員）
        ↓
2. 特徴量エンジニアリング（リーク対策込み）
        ↓
3. 時系列分割（学習 / 検証）
        ↓
4. 来店予測モデル学習
        ↓
5. 来店予測 → 客単価モデル → 売上予測
        ↓
6. 精度評価（ベースライン比較・繁忙日誤差・特徴量重要度）
        ↓
7. LLMプロンプトで結果を解釈・施策提案（自然言語アウトプット）
```

## 9.5 フェーズ7（任意）：継続学習と施策の自動改善

定期実行（日次バッチ等）の中で、システムが「使うほど賢くなる」仕組みを検討する。
**用途を2系統に分けて考えるのが重要**で、適した技術が異なる。

### A. 予測精度の向上 → 強化学習ではなく「継続学習」

来店・売上予測は「翌日に正解が判明する」**教師あり学習**のため、強化学習は不向き。
精度を維持・向上させる正攻法は次の通り。

- **定期再学習（リトレーニング）**：新しい実績を蓄積し、モデルを学習し直して季節・トレンド変化に追従
- **データドリフト検知**：予測のずれを監視し、再学習や見直しのトリガーにする
- **オンライン学習**（任意・高度）：データ到着のたびに少しずつ更新

### B. 施策提案の改善 → 強化学習／文脈付きバンディットが適合

施策は「行動を選び、結果（報酬）から良い選び方を学ぶ」問題であり、強化学習の発想が合う。


| 要素         | 例                              |
| ---------- | ------------------------------ |
| 行動（action） | 発注量、人員シフト、クーポン配信先、値引き率、ポイント倍率日 |
| 報酬（reward） | 粗利、廃棄ロス削減、来店・再来店、会員維持率         |
| 方策（policy） | 状況（天候・曜日・在庫・会員）に応じた最適行動の選び方    |


ただし純粋な強化学習は実務ハードルが高い（探索コスト・報酬遅延・非定常）。
**現実的な落とし所は「文脈付きバンディット（Contextual Bandit）」**で、
クーポン・オファー最適化を、損失を抑えつつ学習する用途に適する。

### 前提条件：効果測定ログ

A・Bいずれも、**「施策を実行した結果どうだったか」を記録する効果測定の仕組み** が前提。
現MVPは「予測→提案」までで実行結果ログが無いため、フェーズ6（DB連携・基盤）と密接に関係する。

### 推奨ステップ（段階的導入）

1. 効果ログ基盤（提案・実行・結果の記録）
2. 予測の再学習ループ＋ドリフト検知（＝強化学習ではない）
3. A/Bテスト・効果測定で施策の良し悪しを可視化
4. 文脈付きバンディットでクーポン・オファー最適化（実用的な入口）
5. 本格的な強化学習（在庫・人時の動的最適化など）はデータ・安全性が整ってから慎重に

> デモ候補：①クーポン最適化の文脈付きバンディット、②再学習＋ドリフト検知ループ。

## 10. 成功基準（MVPのゴール）

- 擬似データに埋め込んだ来店の傾向を、モデルが学習・再現できている
- 単純ベースラインより機械学習の来店予測精度が有意に良い
- 来店予測から売上予測へ妥当に接続できている
- 予測時点で入手可能な情報のみでリークなく予測できている
- 結果をLLMプロンプトで分かりやすく解釈・施策提案できる
- 本番（実データ）設計に向けた課題・必要データが洗い出せる

## 11. 確認事項（オープンクエスチョン）

### 決定済み

- 予測の粒度：**両方**（店舗単位・日次を主軸 → 会員個人は後続フェーズ）
- MVPの成果物：**分析パイプライン（ノートブック）＋ LLM施策提案プロンプト**（擬似データで完成済み）
- データソース判明：購買＝**CSPM**、来店＝**FLOW_COUNTER 入退室イベント**、天候＝**FLOW_COUNTER_DAILY_WEATHER**（提供テーブル定義より）
- 実データは **TSV（タブ区切り）**。購買は187店・2024-10〜2026-06、通行人は単店・2025-11〜2026-06。

### 要判断（実データ検証を進める前の重要決定）★

1. **来店↔購買の店舗リンク**：通行人カウント端末（1台）が CSPM の187店のどれか＝**端末→店舗マッピング**を提供いただけるか。無い場合は A系統（購買）と B系統（通行人）を**別々**に検証する方針でよいか。
2. **実データフェーズの主目的変数**：A（購買から店舗×日の購買客数・売上）を本命に据える方針でよいか。
3. **天候の実データ**：`FLOW_COUNTER_DAILY_WEATHER` の**実物CSVエクスポート**をいただけるか（現状は定義のみ）。翌日予測用の**予報値**取得可否も。
4. **対象店舗の絞り込み**：187店の中から検証対象を選ぶか（データが連続している店舗を優先）、全店で汎用モデルにするか。

### 未確定（従来からの継続確認）

5. **SNS データ**：提供テーブルに無い → 別途取得 / 当面なし / 将来追加 のいずれにするか
6. **FLOW_COUNTER 集計ルール**：`IN` のみで来店数とするか、`UNKNOWN` 端末の扱い、欠損日補完、営業時間の扱い
7. **CSPM**：返品(RETN)の控除ルール（純売上・純客数の定義）、会員/非会員（`V_ACCOUNT_ID` が null）の扱い
8. **カテゴリ分類**：`CSPM_PRODUCT` の大/中/小分類を説明変数化する際の粒度
9. 「人流予測」（外部の周辺人出データ）を別途入手するか（提供データには無い）
10. 追加提案データ（給料日・販促・クーポン等）の実データ有無・採否
